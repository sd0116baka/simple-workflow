## 主流程

任务真源：提供结构化的原始任务。
  ↓
任务解析器：对结构化任务进行标准化、拆分、配置补全和校验，生成任务上下文包。
  ↓
任务池：保存任务上下文包，并维护任务状态、依赖关系、执行记录和审查记录。
  ↓
执行准入器(启动检查)：在推荐任务之前，判断当前是否可以启动新任务，例如是否已有任务运行、仓库状态是否可用。
  ↓
任务推荐器：一次性Agent，从可执行任务中推荐当前最优先的任务，并请求追加执行意图。
  ↓↓
执行意图：任务推荐器请求追加的结构化产物，包含推荐任务标识、推荐理由、预期范围和风险判断。
  ↓
执行准入器(授权追加)：在执行意图已追加到任务上下文包后，判断该包是否可以进入执行阶段；通过则请求追加执行授权，未通过则请求追加准入拒绝或升级人工决策。
  ↓↓
执行授权：确认该任务被允许进入执行阶段，固化了任务目标、验收包、硬执行边界、工作区和终止条件。
  ↓
主 Agent：任务生命周期的管理者，持有执行授权，并调度执行 Agent 和 Review Agent。
  ↓
执行 Agent：每轮执行前由主 Agent 调用命令产生，负责修改、测试并返回执行结果；返回后销毁。
  ↓↓
执行结果：执行 Agent 每轮请求追加的结构化产物，包含修改摘要、变更文件、测试结果、未解决问题和风险说明。
  ↓
Review Agent：每轮执行后由主 Agent 调用命令产生，负责审查执行结果并返回 Review 结果；返回后销毁。
  ↓↓
Review 结果：Review Agent 每轮请求追加的结构化产物，包含审查结论、问题、严重程度、证据和修正建议。
  ↓
收敛 / 修正：每轮 Review 后判断是否收敛；未收敛且未达到终止条件时，主 Agent 继续调度新一轮执行和 Review。
  ↓
人工决策：收敛时，人工决定 accept / commit / closeout；不能收敛时，人工决定取消任务、提供新方案、继续迭代或转人工处理。

### 流程图例解释

↓  表示流程切换，进入下一个环节。
↓↓ 表示当前环节产生关键结构化产物，并请求追加到任务上下文包中，由后续环节消费。

### 流程名词解释

任务上下文包：从任务解析器输出开始的唯一流转对象。后续环节不生成彼此割裂的新对象，而是在同一个任务上下文包中追加执行意图、执行授权、执行结果、Review 结果、收敛判断和人工决策，该对象只能由任务池管理，其他环节通过api请求进行读写。

Project Profile：每个项目独有的结构化策略配置，用于定义默认参数、测试命令、风险规则、允许范围、禁止范围、最大轮次、Review 策略和 Agent 提示词。结构化策略用于硬约束，prompt(type).md 用于定义 Agent 行为提示。

配置补全：根据 Project Profile 中的默认值补齐任务上下文包里的配置型空值。它只补数值化、策略化、配置化字段，不补任务目标、任务特定验收标准等业务语义字段。

可执行任务：已经通过解析和校验、依赖已满足、状态允许执行、且不需要人工澄清的任务。

执行意图：任务推荐器请求追加的结构化建议，表示“建议现在执行这个任务”。它不是执行许可，其中的预期范围是软约束，只用于帮助后续 Agent 聚焦。

执行准入器：负责执行前的确定性门槛。推荐前，它从任务池中请求可启动任务视图，供任务推荐器消费；推荐后，它读取已追加执行意图的任务上下文包，确认该执行意图仍然有效，并判断当前任务包、运行环境和 Project Profile 策略是否足以固化执行授权。执行准入器不解释自然语言，不修改任务业务语义，只请求追加执行授权、准入拒绝或人工决策请求。

执行授权：准入通过后追加到任务上下文包中的固定执行依据，包含任务目标、验收包、硬执行边界、工作区和终止条件。

验收包：判断任务是否完成的检查集合，由任务特定验收标准、Project Profile 中的通用质量门槛，以及任务类型默认检查组成。

执行边界：本次任务允许和禁止的执行范围，主要来自任务上下文包与 Project Profile。执行意图中的预期范围只作为软约束，不构成硬边界。

收敛：任务结果满足验收包，且 Review 结果没有阻塞性问题，可以进入人工 accept / commit / closeout。
